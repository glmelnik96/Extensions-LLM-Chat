# Промпт этапа 3: Внедрение Multi-pass pipeline

**Назначение**: Задать контекст и задачи для внедрения реального multi-pass (generator → validator-1 → validator-2 → repair при необходимости) поверх каркаса, собранного на этапе 2. Использовать вместе с shared-project-context.md и docs/legacy-archive-on-user-request-only/planning/plan-staged-implementation-stages-2-through-4.md (этап 3).

---

## Контекст

- Этап 2 выполнен: handleSend вызывает каркас pipeline; внутри — один вызов LLM с текущей логикой. Вызов облака и сбор сообщений выделены и переиспользуемы.
- Цель этапа 3: внутри каркаса выполнять несколько проходов: generate (gpt-oss-120b) → validate-1 (gpt-oss-120b) → validate-2 (gpt-oss-120b) → при провале валидации repair (Qwen/Qwen3-Coder-Next). В чат и в session попадает только финальный результат; промежуточные стадии отображаются в статусной строке.

---

## Задачи (кратко)

1. Реализовать в каркасе pipeline последовательность: генерация → валидация 1 → валидация 2; при необходимости — один проход repair. Модели: openai/gpt-oss-120b для generate и validators, Qwen/Qwen3-Coder-Next для repair.
2. Финальное выражение из последнего успешного прохода записывать в session.latestExtractedExpression; в messages добавлять один assistant message с финальным текстом (формат: expression + ---EXPLANATION--- + ---NOTES---). Промежуточные ответы моделей в transcript не показывать.
3. Обновлять статусную строку на каждом шаге (например: «Генерация…», «Проверка 1…», «Проверка 2…», «Исправление…», «Готово.»). Сохранить isRequestInFlight на время всего multi-pass; в finally сбрасывать и обновлять кнопки.
4. Apply остаётся только ручным; протокол host и getResolvedTarget() без изменений. Обеспечить совместимость: сессии без pipeline или старые сессии открываются и работают (см. docs/runtime-state-schema.md).
5. Definition of done: Send запускает полный цикл; в чат — один финальный ответ; Apply применяет финальное выражение; старые сессии не ломаются.

---

## Что здесь не делаем

- Не вводим auto-apply.
- Не меняем формат финального сообщения в чате.
- Не подключаем три проекции knowledge base (это этап 4); при необходимости использовать текущий единый docs context для всех проходов.
