# Целевая архитектура — Multi-pass pipeline и Knowledge Base

> **Актуальный продукт (2025–2026):** панель работает как **AE Motion Agent** с циклом tool calling. См. **[capabilities-and-roadmap.md](capabilities-and-roadmap.md)** и **[final-architecture.md](final-architecture.md)**.  
> Этот документ сохраняет **целевое описание исторического multi-pass pipeline** (expression + KB + manual Apply) для справки и возможного reuse кода в `pipelineAssembly.js` / смежных модулях.

Документ ниже фиксирует целевое состояние проекта после внедрения пайплайна и локальной knowledge base в логике «один Send → multi-pass → один финальный ответ». Реализация того pipeline в текущем **main.js** **не активна**; агент — отдельный путь.

---

## 1. Оркестрация

- **Место**: оркестрация pipeline остаётся в **main.js** (или в выделенном модуле, подключаемом из main.js). Внешний backend для оркестрации не вводится.
- **Идея**: один последовательный async flow в панели: пользователь нажимает Send → один «запрос» с точки зрения пользователя → внутри выполняется multi-pass (generate → validate-1 → validate-2 → при необходимости repair) → в чат и в session попадает только финальный результат.

---

## 2. Роли моделей

- **openai/gpt-oss-120b**: используется как **generator** (первый проход), **validator-1** и **validator-2** (проверка выражения/ответа на соответствие правилам и документации).
- **Qwen/Qwen3-Coder-Next**: используется как **repair-model** — отдельный проход только при необходимости исправления (например, после провала валидации). Не подменяет generator по умолчанию.
- **Runtime fallback**: переключение на другую модель только при ошибке модели/сети (например, недоступность gpt-oss-120b → одна попытка с Qwen в рамках того же пользовательского «Send»), без изменения ролей generator/validator/repair в успешном сценарии.

---

## 3. Apply и вывод пользователю

- **Apply**: только **manual**. Кнопка Apply Expression по-прежнему применяет `session.latestExtractedExpression` к выбранному target через host. Авто-apply не вводится.
- **Вывод в чат**: пользователю показывается **только финальный result** (одно сообщение ассистента в формате expression + ---EXPLANATION--- + ---NOTES---). Промежуточные стадии (черновики, результаты валидации, репар) в transcript не выводятся.
- **UI status**: текущая стадия pipeline отображается в **статусной строке** (например: «Генерация…», «Проверка 1…», «Проверка 2…», «Исправление…», «Готово.»). Элементы status line (status-text, model-status) остаются; расширяется только текст/значения, передаваемые в updateStatus/аналог.

---

## 4. Локальная Knowledge Base

- **Источники**: локальная knowledge base собирается из **Adobe Help Center** как primary source и **docsforadobe** как secondary mirror. Один общий корпус документов/сниппетов.
- **Три проекции знаний**: поверх одного корпуса существуют **три отдельные проекции** (роль-специфичные выборки или форматирование):
  - для **generator** (генерация выражения по запросу и контексту);
  - для **validator** (проверка соответствия API и правилам);
  - для **repair** (исправление с опорой на документацию и замечания валидации).
- Реализация проекций может быть: разными запросами к одному индексу, разными системными инструкциями по использованию одних и тех же сниппетов, или отдельными «видами» на один и тот же AE_DOCS_INDEX (или преемник). Текущие модули aeDocsIndex / aeDocsRetrieval / aePromptContext остаются точкой расширения без обязательной замены в первом шаге.

---

## 5. Runtime state и session

- **Session**: существующие поля сессии сохраняются (id, title, createdAt, updatedAt, model, messages, latestExtractedExpression). Добавляется **сокращённое** поле для pipeline (например, `session.pipeline`), только для нужд нового flow, без дублирования всей истории проходов в сообщениях.
- **Схема session.pipeline**: минимальная, совместимая со старыми сессиями — см. docs/runtime-state-schema.md. Старые сессии без поля pipeline считаются single-pass; новый код не ломает загрузку и отображение.

---

## 6. Технические границы

- **CEP panel + host ExtendScript + cloud LLM**: базовая схема не меняется. Все вызовы моделей по-прежнему с панели на облачный endpoint; host только для comp summary и apply expression.
- **Один облачный endpoint**: по-прежнему `https://foundation-models.api.cloud.ru/v1/chat/completions` (или конфигурируемый аналог); смена модели — через параметр `model` в теле запроса.
- **Обратная совместимость**: старые сессии и старый путь «один запрос — один ответ» остаются рабочими до явного переключения на новый pipeline (или новый pipeline включается по флагу/настройке с сохранением возможности отката).
