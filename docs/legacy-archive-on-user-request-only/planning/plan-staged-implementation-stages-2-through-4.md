# План поэтапного внедрения Multi-pass pipeline и Knowledge Base

Три этапа после завершения аудита (этап 1). Каждый этап имеет scope, ограничения «что нельзя менять», definition of done, риски отката и зависимости.

---

## Этап 2: Инфраструктурный безопасный рефакторинг и каркас pipeline

**Цель**: подготовить код к multi-pass без изменения пользовательского поведения. Текущий single-pass Send и Apply работают как раньше.

**Scope**:
- Выделить из main.js логику вызова LLM (и при необходимости сборки сообщений/docs/target) в функции или модуль, которые можно вызывать многократно с разными ролями/промптами (например, `callCloudChat(session, messages, model, options)`).
- Ввести **каркас** pipeline: объект или набор функций (например, `runPipeline(session, userMessage)`), внутри которого пока что выполняется ровно один вызов — текущий callOllamaForSession по сути. Точка входа handleSend переключается на вызов этого каркаса.
- Сохранить единую точку извлечения выражения: результат каркаса по-прежнему обрабатывается через extractExpressionFromResponse → session.latestExtractedExpression; в чат добавляется один assistant message.
- При необходимости — вынести константы (DEFAULT_MODEL, FALLBACK_MODEL, CLOUD_API_*), списки моделей, формат статусов в отдельные места для последующего использования в этапах 3–4.

**Что нельзя менять**:
- Поведение пользователя: один Send → один ответ в чате, один блок expression + ---EXPLANATION--- (+ ---NOTES---).
- Формат сессии в localStorage (добавлять новые опциональные поля можно; удалять или менять смысл существующих — нельзя).
- handleApplyExpression, getResolvedTarget(), протокол host (applyExpressionToTarget), refreshActiveCompFromHost.
- Логику isRequestInFlight, updateSendEnabled, updateApplyExpressionEnabled, отображение status line.

**Definition of done**:
- handleSend вызывает единый «pipeline» (каркас), который внутри выполняет текущий single-pass flow.
- Все тесты сценария «Send → ответ → Apply» проходят так же, как до рефакторинга.
- Код готов к подстановке нескольких шагов (generate / validate / repair) на этапе 3 без переписывания handleSend и без дублирования логики сборки сообщений.

**Rollback**: откат — вернуть в handleSend прямой вызов callOllamaForSession и убрать промежуточный слой каркаса; риск низкий при условии, что каркас не меняет формат сообщений и состояния.

**Зависимости**: нет блокеров; можно начинать после этапа 1.

## Этап 3: Внедрение нового Multi-pass pipeline

**Цель**: включить реальный multi-pass (generator → validator-1 → validator-2 → при необходимости repair) поверх готового каркаса. Пользователь по-прежнему видит один финальный ответ; промежуточные стадии — только в UI status.

**Scope**:
- В каркасе pipeline: последовательно выполнять генерацию (gpt-oss-120b), затем валидацию 1 и 2 (gpt-oss-120b), при провале валидации — один проход repair (Qwen/Qwen3-Coder-Next). Финальный результат — одно выражение, записывается в session.latestExtractedExpression; в messages добавляется один assistant message с финальным текстом.
- Статусная строка: обновлять текст на каждом шаге (например, «Генерация…», «Проверка 1…», «Проверка 2…», «Исправление…», «Готово.»).
- Сохранить ручной Apply; не вводить auto-apply.
- При ошибке модели/сети — использовать существующую идею fallback (один повтор с FALLBACK_MODEL в рамках того же «Send»), без усложнения ролей.

**Что нельзя менять**:
- Apply только manual; протокол host и getResolvedTarget() без изменений.
- Формат финального сообщения в чате (expression + ---EXPLANATION--- + ---NOTES---).
- Совместимость старых сессий: сессии без pipeline или с пустым pipeline должны открываться и работать (single-pass или отображение последнего результата).

**Definition of done**:
- При нажатии Send выполняется полный цикл generate → validate-1 → validate-2; при необходимости repair; в чат попадает один финальный ответ; latestExtractedExpression соответствует финальному выражению; Apply применяет его.
- В статусной строке поочередно отображаются стадии.
- Старые сессии не ломаются (загрузка, отображение, Apply по последнему выражению).

**Rollback**: откат — переключить pipeline на «single-pass only» (флаг или замена реализации каркаса на одну ступень); риск средний, нужен явный переключатель или ветка кода.

**Зависимости**: этап 2 выполнен (каркас и выделенные вызовы LLM готовы).

## Этап 4: Подключение локальной Knowledge Base и role-specific prompt system

**Цель**: использовать локальную knowledge base (Adobe Help Center + docsforadobe) и три проекции знаний для generator, validator и repair.

**Scope**:
- Один корпус: primary source — Adobe Help Center, secondary — docsforadobe (формат и место хранения/индекса могут быть расширением текущего aeDocsIndex или отдельным индексом).
- Три проекции: для generator, validator и repair — разные запросы к корпусу и/или разное форматирование/инструкции по использованию одних и тех же сниппетов (или три «роль-специфичных» набора правил выбора из корпуса).
- Интеграция в pipeline: на каждом шаге (generate, validate-1, validate-2, repair) использовать соответствующую проекцию для сборки system/context сообщений (аналог текущего BUILD_DOCS_CONTEXT_MESSAGE с разными ролями).
- Не менять базовый формат ответа пользователю и ручной Apply.

**Что нельзя менять**:
- Apply только manual; один финальный вывод в чат; общая оркестрация в main.js (или выделенном модуле панели).
- Совместимость сессий и формат session.pipeline по runtime-state-schema.

**Definition of done**:
- Generator, validator и repair получают контекст из knowledge base через свои проекции.
- Документация по сбору корпуса и формату проекций зафиксирована; при отсутствии данных панель деградирует до поведения без docs (как сейчас при пустом индексе), без падений.

**Rollback**: откат — отключить использование проекций (передавать пустой или старый единый контекст); риск низкий при модульной подаче контекста.

**Зависимости**: этап 3 выполнен; желательно наличие спецификации формата корпуса и проекций (может быть частью этапа 4).
